# 当前问题v1.1
目前该项目已经能够正常运行，但是有一些问题需要改进
## 1.耦合度问题
首先是代码的耦合度太高。当前主要业务逻辑全部集中在 `web_ui/main.py`（长达 900 多行），混合了 FastAPI 路由配置、文件读写、多线程任务调度（Task Queue）、系统进程调用（subprocess 跨语言调用 Node.js）和各种杂项业务处理（如处理 PPT 导出的冗长 JSON 转换等）。
建议的重构方案：
- 引入路由树（Router），按业务模块拆分路由（例如拆分为 `api/paper_router.py`、`api/task_router.py`、`api/config_router.py`、`api/ppt_router.py` 等）。
- 将具体的业务逻辑抽离到 `services/` 层，例如 `services/task_runner.py` 专门负责多线程排队调度，`services/file_manager.py` 专门负责文件系统读写。
- 取消目前基于 `subprocess` 硬编码调用独立脚本（如 `universal_kb_builder.py`、`tools/paper_translator.py` 等）的混乱方式，将它们统一重构并作为后端模块引入。

## 2.废弃代码问题
像是 `E:\workspace\ddl\standalone_pdf2ppt\llm_client.py` 
其中用于剧本生成的代码（例如 `process_paper_two_stage`、`process_paper_three_stage` 以及 `chat_with_character` 中涉及到的“Galgame”、“人设交互”、“脑际连接”等逻辑）和本项目毫无关系，需要被彻底清理删除。仅需保留并优化用于学术报告分析生成的 `get_stage1_md` 等核心解析逻辑。
此外，项目中可能还存在其他早期跑通跑测试时遗留的“胶水代码”，后续需要一并排查删除。

# 服务端梳理（用于管理端口，可能有遗漏）
目前项目依赖多个独立脚本拼凑而成，重构时需要将这些核心能力整合为标准化的后端服务模块（Service），并对外暴露统一 API：
## 论文解析 (Parse Service)
- **当前实现**：通过 `standalone_pdf2ppt/llm_client.py` 提取文本和图表，并调用大模型（SiliconFlow API）生成 Markdown。
- **重构目标**：提供标准的 `/api/parse` 接口，支持异步状态查询。
## 论文翻译 (Translate Service)
- **当前实现**：调用 `tools/paper_translator.py` 脚本生成 `_translated.pdf`。
- **重构目标**：提供 `/api/translate` 接口，与解析任务解耦。
## 论文标注 (Annotation Service)
- **当前实现**：调用 `tools/pdf_annotator.py` 生成 `_annotated.pdf`。
- **重构目标**：提供 `/api/annotate` 接口，并确保它在论文解析完成后再执行。
## ppt生成 (PPT Service)
- **当前实现**：通过 `subprocess` 调用 Node.js 脚本 `standalone_pdf2ppt/ppt_maker/generate_full_ppt.js`，且 `main.py` 中充斥着用于解析 `.pptx` XML 结构的庞大代码。
- **重构目标**：提供 `/api/ppt/generate` 和 `/api/ppt/export` 接口，前端只需通过接口获取数据用于渲染 PPT 编辑器。
## 知识库问答 (QA Service)
- **当前实现**：基于 `main.py` 中 `simple_rag_search` 的轻量级 TF-IDF / 关键词交集匹配法。
- **重构目标**：提供标准化的 `/api/chat` 接口，后续考虑接入正规的向量数据库以优化 RAG 流程。
